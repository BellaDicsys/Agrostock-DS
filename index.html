Agrostock Design System V1

Esta documentacion ya no describe una UI web generica. Describe una plataforma real construida en Ionic y Angular, con una grilla dinamica no mobile para esta V1, una base de dimensionamiento por DVH, una variante admin con escala reducida y una regla fuerte de composicion por componente. El sistema parte de templates reales y de restricciones concretas, no de teoria abstracta.

Foundation

Premisas estructurales del sistema

La UI se organiza sobre tokens calculados, una unidad base de altura visible y wrappers reales del ecosistema Ionic. Esto condiciona directamente el mapeo de componentes y el modo de documentarlos.

Ionic y Angular

La documentacion debe reflejar componentes reales como app-header, ion-content, ion-item, ion-input, ion-select, app-button, app-filter-panel y app-spinners.

DVH como base

--ux_base_unit: 1dvh; gobierna alturas, radios, gaps, paddings, fuentes y escalas visuales. Esto no es un detalle tecnico menor, es la base de la densidad del sistema.

Escala global

--ux_ui_scale define el peso visual de toda la interfaz. La documentacion debe asumir que un mismo bloque puede verse distinto segun el contexto sin cambiar su semantica.

Templates diagnosticados

Los patrones del DS se construyen desde plantillas reales de pantalla, especialmente flujos con welcome, búsqueda, helper, filtros, loaders, empty states y acciones.

Variante

Admin no es otro sistema, es la misma semántica a otra escala

La seccion admin reduce la escala de interfaz, pero no cambia la logica de composicion. El mismo componente debe conservar su rol, aunque se compacte.

Regla base

app-admin-layout
app-admin-layout { --ux_ui_scale: 0.85 !important; --ux_base_unit: 1dvh !important; @extend :root; }

Impacto real

Densidad
Mas informacion visible en la misma altura.
Menor tolerancia a textos largos dentro de acciones, cards y helper.
Mayor importancia del orden entre helper, contenido y acciones.
Regla mandatoria

SVG icons con fill igual a currentColor

Todo icono SVG usado por el sistema debe permitir heredar color desde CSS. Eso vuelve consistente el uso de estados, temas, hover y clases semánticas como text-main, text-side, text-primary, text-success y text-warn.

Regla correcta

SVG base
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 24 24" fill="currentColor" > <path d="..." /> </svg>

No permitido

Rompe semantica visual
<svg fill="#1a237e">...</svg> <svg fill="#000000">...</svg> <path fill="red" d="..." />

Cuando el SVG trae color duro, deja de responder a clases del sistema y rompe la lógica cromática del componente.

Regla mandatoria

app-button como único elemento de click

Dentro del sistema, todo elemento accionable debe resolverse mediante <app-button>. Esto incluye acciones primarias, icon actions, suffix actions, paginación, filtros, toggles operativos y accesos que visualmente parezcan botones.

Regla de uso

Click centralizado
Usar app-button para toda acción del usuario.
Evitar button, a, div clickable o ion-button directo como interfaz final de interacción.
Beneficio una sola capa de semántica, estilos, estados disabled, ids de testing y manejo de icono y texto.

Ejemplo real

Suffix button
<app-button class="suffix" *ngIf="spreadsheetQuota.controls.quota.value && spreadsheetQuota.controls.quota.value.length > 0" [btnId]="'spreadsheetQuotaClearBtn'" [btnClass]="'ion-btn icon suffix main'" [btnIcon]="globalImages.ux_close" (clicked)="clearInput()" ></app-button>
Patrón clave

Welcome como chapa informativa de sección

En Agrostock, .welcome no es solo empty state. También funciona como chapa inicial de contexto de módulo o de tarea. Puede convivir con búsqueda, loaders y estados sin contenido. Su valor real es introducir la acción que sigue.

Rol funcional

No es decorativo
Contexto de sección explica qué acción va a realizar el usuario.
Entrada calmada reduce brusquedad en pantallas centradas con una sola tarea.
Empty state reutilizable también puede usarse cuando no hay resultados.

Patrón clave

Search bar container como addon singleton por componente

La búsqueda no se interpreta como un bloque global del sistema. Se interpreta como un módulo de entrada asociado a una pantalla o componente concreto. Por eso .search_bar-container funciona como addon singleton: aparece cuando la tarea lo necesita y no compite con el resto del layout.

Lectura correcta

Una pantalla, una búsqueda
Singleton visual normalmente una sola búsqueda principal por pantalla.
Acoplada al flujo puede vivir debajo de welcome o arriba de resultados.
Con acciones internas prefijo, suffix clear, submit y chips o zonas opcionales.

Estructura real

Bloques operativos

Helper, content-view y main-actions

Estos tres bloques sostienen la operativa de resultados. El helper resume, el content-view contiene y el main-actions acciona. En Agrostock conviene conservar esa separación incluso en pantallas chicas o centradas.

Bloque Variantes Uso
.helper .helper, .helper.w-filter Resume, filters CTA, paginación, contexto de resultados
.content-view .content-view, .w-actions, .fluid-height, .center Contenido principal de pantalla
.main-actions con o sin .progress-container Acción primaria, confirmación, guardado o avance

Reglas finales

Criterios de uso para que el sistema no se desordene

Estas reglas cierran la lectura del documento y dejan fijado cómo debe evolucionar el DS sin romper la base del proyecto.

Regla 1

Todo icono SVG del sistema debe tener fill="currentColor".

Regla 2

Todo click del usuario debe resolverse mediante app-button.

Regla 3

welcome se documenta como chapa informativa de sección y también como empty state.

Regla 4

search_bar-container se documenta como addon singleton por componente o pantalla.

Regla 5

El DS debe reflejar templates reales de Ionic y Angular, no ejemplos HTML abstractos sin correspondencia con el proyecto.

Regla 6

Admin es la misma semántica del sistema a una escala visual menor, no otro design system.

Lectura ejecutiva

que significa

Este cambio no es cosmetico. El proyecto elimina mucho mas SCSS del que agrega, con una concentracion muy fuerte en pages y componentes compartidos.

  • La branch reduce peso visual neto, no solo redistribuye archivos.
  • UI pages es la zona mas impactada, señal de simplificacion por pantalla.
  • Global y theme crecen, lo que sugiere centralizacion de reglas comunes.
  • old-button sube y button baja, posible reemplazo, rollback o migracion de componente base.
Insight:

La lectura mas consistente es una poda de estilos legacy con traslado parcial hacia capas globales y utilitarias.
